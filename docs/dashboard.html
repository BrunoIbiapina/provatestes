Princípios de design

// SIX RULES
01 · LAYERING
Page Object Model
Cada tela do SauceDemo virou uma classe que esconde seletores e esperas. Os testes leem como história de usuário.
02 · LAYERING
Service / Client
Cada recurso da API tem seu cliente HTTP. Testes nunca falam direto com requests — sempre via PetClient, StoreClient, UserClient.
03 · DATA
Factories & Faker
Zero hardcode. Cada teste gera massa fresca, com IDs grandes (10⁸–10⁹) para evitar colisão na API pública.
04 · STABILITY
Esperas explícitas
Nenhum time.sleep no projeto. WebDriverWait + ExpectedConditions resolve flakiness na origem.
05 · CONTRACT
Schema validation
Toda resposta de sucesso da API é validada contra um JSON Schema. Mudança de contrato é detectada antes de produção.
06 · HYGIENE
Cleanup automático
Fixtures usam yield: o teardown roda mesmo quando o teste falha. Nada de lixo na base.
